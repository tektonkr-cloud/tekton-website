The Problem

BAS는 제어합니다.
하지만 데이터를 검증하지 않습니다.

센서가 고장나도 SCADA 화면엔 '정상'이 뜹니다.
그 사이 에너지는 매달, 조용히 새어나갑니다.

01 / 센서 드리프트

온도센서가
5°C 높게 표시됩니다.

AHU 급기 덕트 온도센서 경년열화. 실제 22°C → 센서 27°C 표시. 냉방 밸브 과다 개방. BAS 화면엔 "정상".

↑ 월 40~60만원 초과 냉방비
02 / 통신 두절

60개 포인트가
수개월째 고착입니다.

RS-485 배선 노후, 접촉 불량. 센서 60개 데이터 고착. 서버엔 마지막 정상값 표시. 평균 2~6개월 후 발견.

↑ 수동 운전, 관리 공백 지속
03 / 밸브 고착

DDC는 닫혔다 하지만
실제론 열려 있습니다.

FCU 냉온수 밸브 액추에이터 고착. DDC 닫힘(0%) 명령 → 실제 30% 열림. 겨울철 냉수 지속 유입.

↑ 월 80~120만원 에너지 낭비
T-ARCH Solution

FDD가 아닙니다.
포렌식입니다.

FDD는 센서를 신뢰하고 룰을 적용합니다. T-ARCH는 센서 값이 맞는지 먼저 검증합니다.

기존 FDD / BAS

기존 방식
✕센서 값을 정확하다고 전제하고 룰 적용
✕임계값 초과 시 알람 — 이미 낭비 발생 후
✕드리프트된 센서로 제어 → 오제어 지속
✕에너지 낭비 "추정" 리포트
✕제조사 종속, 교체 비용 수천만~수억 원

STEP 01

센서 무결성 검증

고착(Stuck), 급변(Spike), 드리프트, 범위 이탈을 실시간 탐지합니다.

STEP 02

통신 건전성 분석

RS-485 패킷 손실, 타임스탬프 불일치, 데이터 고착을 자동 감지합니다.

STEP 03

에너지 이상패턴

동시냉난방, 비가동 시간 운전, 낭비 금액을 원 단위로 정량화합니다.

STEP 04

제어 명령 검증

DDC 명령값 vs 실측값 불일치, 인터록 위반을 자동 탐지합니다.

선 하나 없이.

⚡ 30분 이내 연동 완료

BACnet / IP
REST API 연동

BACnet/IP 네트워크가 구성된 현장에서 가장 빠른 방식입니다. 물리 배선 없이 아래 세 가지 중 하나만 제공하시면 됩니다.

  • ARead-Only 계정 생성 후 IP/포트 전달
  • BDDC 포인트 맵(Object List) 파일 제공
  • CREST API 엔드포인트 오픈 (IP + 포트 허용)
  • 4T-ARCH 자동 탐지 → 포렌식 시작

BACnet MS/TP
멀티드롭 버스 연동

BACnet MS/TP 기반 현장에서 기존 버스에 직접 연결합니다. 네트워크 설정만으로 완료됩니다.

  • 1BACnet 네트워크 구성 정보 확인 (Device ID 등)
  • 2T-ARCH 서버 IP를 네트워크에 등록
  • 3Who-Is / I-Am으로 Device 자동 탐지
  • 4Object List 읽기 → 포렌식 시작

RS-485
패시브 버스 탭

구형 현장 또는 Niagara 미설치 현장에서 RS-485 멀티드롭 버스에 패시브 탭(T분기)으로 연결합니다.

  • 1RS-485 버스 A/B 단자에 패시브 탭 연결

  • 3Baud rate, Slave 주소 설정
